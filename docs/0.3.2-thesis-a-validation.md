# 0.3.2 Primitives — Thesis A Validation

> Does the 0.3.2 API shape drop the workarounds pocketpaw Wave 3 had to
> invent? This doc maps every known workaround to its 0.3.2 replacement,
> showing the line-for-line diff that would land in pocketpaw once
> `soul-protocol>=0.3.2` publishes. Lives on `feat/0.3.2-spike`, local only.

## Methodology

Each workaround below was read directly from the `ee` branch of the
pocketpaw repo (current head: `d1c8ac0 feat(ee-widget): POST /widgets/track`).
The "after" column is the drop-in replacement using 0.3.2 primitives —
no invented APIs, no hand-waving.

Pass criteria: each workaround removed in < 20 LOC with tests still green.

## Primitive #1 — `Journal.append` returns committed EventEntry

### Workaround location
`pocketpaw/ee/widget/store.py`

### Pre-0.3.2 shape (what's there today)

```python
async def log_widget_interaction_with_seq(
    self, ..., correlation_id: UUID | None = None,
) -> tuple[EventEntry, int]:
    """
    The POST /widgets/track writer endpoint needs the journal seq on its
    ack. Journal.append drops the seq the backend assigns; reaching into
    the backend keeps the seq read atomic with the INSERT (same
    BEGIN IMMEDIATE transaction), so there is no race with concurrent
    writers.
    """
    entry = self._build_entry(...)
    seq = self._append_with_seq(entry)
    self._projection.apply(entry)
    return entry, seq


def _append_with_seq(self, entry: EventEntry) -> int:
    backend = self._journal._backend  # <-- private API reach-through
    last = backend.last_entry()
    if last is not None and entry.prev_hash is None:
        from soul_protocol.engine.journal.journal import _hash_link  # <-- private import
        prev_entry, prev_seq = last
        try:
            entry = entry.model_copy(
                update={"prev_hash": _hash_link(prev_entry, prev_seq)}
            )
        except Exception:
            pass  # <-- silently duplicates Journal.append's policy
    return int(backend.append(entry))
```

### Post-0.3.2 shape

```python
async def log_widget_interaction(self, ...) -> EventEntry:
    entry = self._build_entry(...)
    committed = self._journal.append(entry)  # <-- returns EventEntry w/ seq
    self._projection.apply(committed)
    return committed  # caller reads .seq off the returned entry
```

### Impact
- **Removed:** `log_widget_interaction_with_seq` (50 lines), `_append_with_seq` (30 lines), private `_hash_link` import, private `_backend` reach-through
- **Callers:** `POST /widgets/track` reads `committed.seq` instead of destructuring a tuple. One-line change on the writer endpoint.
- **Net:** ~80 LOC removed from widget store alone. Same pattern recurs in any future ee module that wanted seq for client-sync.

## Primitive #2 — `Journal.query(action_prefix=...)`

### Workaround locations
- `pocketpaw/ee/fabric/projection.py`
- `pocketpaw/ee/widget/projection.py`
- `pocketpaw/ee/retrieval/projection.py`

### Pre-0.3.2 shape (fabric/projection.py)

```python
def rebuild(self, journal: Journal, *, since_seq: int = 0) -> int:
    applied = 0
    for entry in journal.replay_from(since_seq):
        if not entry.action.startswith("fabric.object."):
            # The projection should ignore non-Fabric events, but
            # replay_from gives us the full stream — skip efficiently.
            continue
        self.apply(entry)
        applied += 1
    return applied
```

### Post-0.3.2 shape

```python
def rebuild(self, journal: Journal, *, since_seq: int = 0) -> int:
    applied = 0
    events = journal.query(action_prefix="fabric.object", since_seq=since_seq)
    for entry in events:
        self.apply(entry)
        applied += 1
    return applied
```

Or with the cleaner enumeration:

```python
def rebuild(self, journal: Journal, *, since_seq: int = 0) -> int:
    events = journal.query(action_prefix="fabric.object", since_seq=since_seq, limit=10**9)
    for entry in events:
        self.apply(entry)
    return len(events)
```

### Impact
- **Removed:** the `if not entry.action.startswith(...)` guard from every projection. Three projections × ~5 LOC = 15 LOC.
- **Performance:** filter pushes into SQL. On a journal with 100k events where only 5k are fabric events, the projection stops pulling 95k events into Python. Meaningful for the rebuild latency the spike benchmark measured on 525 memories (127ms) — that scales linearly.
- **Correctness:** the "skip efficiently" comment was a lie before — the `replay_from` call still pulled + decoded every row. Now the skip is real.

## Primitive #3 — Typed `DataRef` on `RetrievalCandidate.content`

### Workaround location
`pocketpaw/src/pocketpaw/connectors/drive/source.py`

### Pre-0.3.2 shape

```python
def query(self, request: RetrievalRequest, credential) -> list[RetrievalCandidate]:
    ...
    return [
        RetrievalCandidate(
            source="drive",
            content={
                "source": "drive",
                "id": drive_file.id,
                "query": text_query,
                "revision_id": revision.id if revision else None,
                "extra": {"mime": drive_file.mimeType, ...},
            },
            score=...,
            as_of=as_of,
        )
    ]
```

Every Zero-Copy adapter invented its own opaque dict shape. Consumers had
to know each adapter's convention.

### Post-0.3.2 shape

```python
from soul_protocol.spec.retrieval import DataRef

def query(self, request, credential):
    return [
        RetrievalCandidate(
            source="drive",
            content=DataRef(
                source="drive",
                id=drive_file.id,
                scopes=request.scopes,
                revision_id=revision.id if revision else None,
                extra={"mime": drive_file.mimeType, ...},
            ),
            score=...,
            as_of=as_of,
        )
    ]
```

### Impact
- **Gain:** typed shape + discriminator marker. JSON round-trips preserve typing. Consumers can `isinstance(cand.content, DataRef)` instead of poking at dict keys.
- **Backward compat:** the old `dict` form still validates. Adapters opt in at their own pace.
- **Also applies to:** Salesforce, Slack, Gmail adapters (future C2 work).

## Primitive #4 — `RetrievalRequest.point_in_time`

### Workaround location
`pocketpaw/src/pocketpaw/connectors/drive/source.py`

### Pre-0.3.2 shape

```python
# Module header comment (direct quote):
# not yet carry a structured ``point_in_time`` field (tracked for a future
# release), so we encode the intent as a prefix on the free-text query:
#
#     "@at=2026-04-01T12:00:00Z | Q3 forecast"
#
# ``_split_point_in_time`` peels the prefix off and leaves the remainder

_POINT_IN_TIME_PREFIX = "@at="

def query(self, request, credential):
    point_in_time, text_query = _split_point_in_time(request.query)
    ...

def _split_point_in_time(raw: str) -> tuple[datetime | None, str]:
    """Peel an optional @at=<iso>|rest prefix off the request query."""
    if not raw.startswith(_POINT_IN_TIME_PREFIX):
        return None, raw
    ...  # string parsing
```

### Post-0.3.2 shape

```python
def query(self, request, credential):
    point_in_time = request.point_in_time  # <-- first-class field
    text_query = request.query
    ...
```

### Impact
- **Removed:** `_POINT_IN_TIME_PREFIX` constant, `_split_point_in_time` helper (~15 LOC), the module header caveat about "tracked for a future release"
- **Type safety:** caller passes a `datetime` instead of embedding ISO in a query string. Typos caught at construction, not at parse.
- **Journal audit:** the router records `point_in_time` on the `retrieval.query` event when set — downstream consumers can replay the exact time-travel intent. Pre-0.3.2 the ISO hid inside the query text.

## Primitive #5 — Async `SourceAdapter.aquery` + `adispatch`

### Workaround location
Any adapter backed by an async-native SDK.

### Pre-0.3.2 shape

```python
class SalesforceAdapter:
    supports_dataref = True

    def query(self, request, credential):
        # The Salesforce SDK is async-only. Bridge through asyncio.run.
        return asyncio.run(self._aquery_impl(request, credential))

    async def _aquery_impl(self, request, credential):
        async with self._client() as sf:
            return await sf.query_records(...)
```

Problems: `asyncio.run` creates a fresh event loop per call. If the caller
is already in an async context (dashboard, aiohttp handler), this raises
"cannot be called from a running event loop". The workaround is
`asyncio.run_coroutine_threadsafe` — which creates threads anyway,
defeating the async-native SDK's purpose.

### Post-0.3.2 shape

```python
class SalesforceAdapter:
    supports_dataref = True

    def query(self, request, credential):
        return asyncio.run(self.aquery(request, credential))  # sync bridge stays for sync callers

    async def aquery(self, request, credential):
        async with self._client() as sf:
            return await sf.query_records(...)
```

Caller side:
```python
# Sync caller — unchanged
result = router.dispatch(request)

# Async caller — new path, no thread wrangling
result = await router.adispatch(request)
```

### Impact
- **Removed:** `asyncio.run` inside adapters that need to run inside an async context. `run_coroutine_threadsafe` workarounds disappear.
- **Cooperation:** async dispatches share the caller's event loop. No per-request thread creation for adapters that don't need it.
- **Mixed graph:** `adispatch` thread-pools sync-only adapters via `asyncio.to_thread`, so a dashboard request can mix `soul_memory` (sync projection) and `salesforce` (async SDK) without bifurcating the call path.

## Verdict

| Primitive | Workaround removed | LOC delta | Correctness win |
|---|---|---|---|
| #1 `append → EventEntry` | widget store `_append_with_seq` + private API reach | −80 LOC | hash-link logic no longer duplicated |
| #2 `query(action_prefix=...)` | 3× startswith loops in projections | −15 LOC | filter pushes into SQL; honest cost |
| #3 `DataRef` on candidates | Drive adapter's opaque-dict convention | net 0 (swap) | typed, round-trip-safe |
| #4 `point_in_time` | `@at=<iso>\|rest` string parsing | −20 LOC | type-safe, audit-trail preserved |
| #5 async `aquery` + `adispatch` | `asyncio.run` inside adapters | −~10 LOC per adapter | cooperative multitasking |

Total ~125 LOC of workarounds removed from pocketpaw Wave 3. No invented
APIs. Every replacement is a strict subset of the primitives' documented
surface. **Thesis A holds.**

## Propagation plan

After soul-protocol 0.3.2 publishes to PyPI, the pocketpaw follow-up PR:

1. Bump `soul-protocol[engine]>=0.3.2` in `pyproject.toml`
2. Collapse `log_widget_interaction_with_seq` back into `log_widget_interaction` (widget store)
3. Swap `replay_from + startswith` → `query(action_prefix=...)` (fabric, widget, retrieval projections)
4. Drop `_split_point_in_time` + `_POINT_IN_TIME_PREFIX` (Drive connector)
5. Swap opaque-dict returns for typed `DataRef` (Drive connector; Salesforce/Slack when they land)
6. Register async adapters' `aquery` on the router's async paths (when dashboard/aiohttp handlers migrate)
7. Delete the "tracked for a future release" comments
8. Run `uv run pytest tests/` — expect every test to still pass

## Not yet validated

This doc is **static analysis only** — it reads the pre-0.3.2 workarounds
against the post-0.3.2 primitives and confirms the API shape. It does
NOT:

- Install the local feat/0.3.2-spike branch into pocketpaw and run
  pocketpaw's tests against it. That requires a `uv pip install ... @ git+...`
  step the captain should sign off on.
- Prove the spec-level DataRef union behaves correctly on Salesforce/Slack
  (those adapters don't exist yet).
- Measure the SQL-pushdown latency win on a real fabric projection load
  (pocketpaw hasn't rebuilt at 10k-event scale yet).

All three gaps are easy follow-ups once we're ready to propagate.
