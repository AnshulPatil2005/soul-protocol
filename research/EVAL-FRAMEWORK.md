<!-- research/EVAL-FRAMEWORK.md -->\n<!-- Created: 2026-03-11 — Comprehensive 7-dimension eval framework for Soul Protocol -->\n<!-- Purpose: Spec for implementation agents building eval/suite.py and supporting runners -->\n\n# Soul Protocol Evaluation Framework\n\nVersion: 0.1.0\nStatus: Design Spec (ready for implementation)\nAuthors: Architect Agent\nLast Updated: 2026-03-11\n\n---\n\n## Overview\n\nSoul Protocol is not a better RAG system. It is a persistent identity layer —\npsychology-informed memory, emotional intelligence, personality expression, relationship\nmodeling, and portable self-concept. The current research infrastructure (long_horizon/runner.py)\nmeasures one dimension well: memory recall. This document specifies a 7-dimension framework\nthat covers the full value proposition.\n\nThe framework produces a single **Soul Health Score** (SHS) — a 0-100 composite — alongside\nper-dimension breakdowns suitable for whitepapers, benchmarks, and dashboards.\n\n### Design Constraints\n\n1. **No LLM required for core metrics.** Every metric in this framework can be computed\n   by running Soul Protocol's own heuristic pipeline. LLM-dependent metrics are explicitly\n   flagged as \"LLM-enhanced\" and are optional extensions.\n\n2. **Reproducible.** All randomness is seeded. Same seed → same results across runs.\n\n3. **Runs in CI.** The full suite completes in under 5 minutes on a laptop without any\n   API keys. LLM-enhanced runs are separate and optional.\n\n4. **Produces whitepaper numbers.** Every metric has a defined value range, a baseline\n   to compare against, and an effect size calculation.\n\n5. **Extends the existing infrastructure.** The runner.py / scenarios.py / analyze.py\n   pattern from research/long_horizon/ is the model. New dimensions follow the same\n   dataclass/runner/analyzer pattern.\n\n---\n\n## The 7 Dimensions\n\n| # | Dimension | Weight in SHS | LLM Required? |\n|---|-----------|--------------|---------------|\n| 1 | Memory Recall | 20% | No |\n| 2 | Emotional Intelligence | 20% | No (core) / Yes (enhanced) |\n| 3 | Personality Expression | 15% | No (structural) / Yes (output) |\n| 4 | Bond / Relationship | 15% | No |\n| 5 | Self-Model | 15% | No |\n| 6 | Identity Continuity | 10% | No |\n| 7 | Portability | 5% | No |\n\nWeights reflect what a first-time reader of the whitepaper will care most about:\nmemory and emotion are the core claims; portability is table stakes (it works or it doesn't).\n\n---\n\n## Dimension 1: Memory Recall\n\n### What It Measures\n\nThe soul's ability to store significant information and retrieve it on demand —\neven when buried under many unrelated turns.\n\n### Current State\n\nAlready measured in `research/long_horizon/`. This dimension keeps the existing metrics\nand adds two new sub-metrics: noise resistance and storage selectivity.\n\n### Metrics\n\n| Metric | Definition | Target | Baseline |\n|--------|-----------|--------|---------|\n| `recall_precision` | Fraction of test queries where expected content appears in top-10 recall results | >= 0.80 | RAG-only: ~0.55 |\n| `storage_rate` | memories stored / total turns | 0.20-0.40 (selective) | RAG-only: ~1.0 (everything) |\n| `burial_recall` | recall_precision specifically on adversarial_burial scenario (140-turn noise) | >= 0.70 | RAG-only: expected ~0.30 |\n| `fact_hit_decay` | Recall hit rate for facts planted at turn 0-10 vs facts planted at turn 100+ in the same scenario | Decay < 15% | RAG-only: expected > 40% decay |\n\n### Scenarios\n\nUse the three existing `long_horizon/` scenarios: `life_updates`, `emotional_rollercoaster`,\n`adversarial_burial`. The `adversarial_burial` scenario is the primary driver for `burial_recall`.\n\n### Scoring (0-100)\n\n```\nscore = (recall_precision * 50) + ((1 - storage_rate_excess) * 20) + (burial_recall * 30)\n\nwhere storage_rate_excess = max(0, storage_rate - 0.40)\n```\n\nRationale: precision matters most (50 pts), burial recall proves the significance gate\ndoes real work (30 pts), storage selectivity proves it isn't just storing everything (20 pts).\n\n### Implementation Notes\n\n- Code path: `research/long_horizon/runner.py` → `_run_condition()` → `soul.recall()`\n- The runner already collects `recall_hits`, `recall_misses`, `total_memories`, `episodic_count`\n- Add `burial_recall` by running `per_scenario_breakdown()` filtered to `adversarial_burial`\n- `fact_hit_decay` requires tagging test_points with their planting turn depth — add a\n  `depth_bucket` field to `TestPoint` (\"early\" = planted in first 20% of turns, \"late\" = last 20%)\n\n---\n\n## Dimension 2: Emotional Intelligence\n\n### What It Measures\n\nWhether the somatic marker system (Damasio) actually works: does the soul correctly detect\nemotional tone, attach accurate markers to memories, update its mood state in response,\nand prioritize emotional memories for recall?\n\n### Why This Matters\n\nEmotional intelligence is what separates a soul from a database. A RAG system stores\n\"User: I lost my dog\". A soul stores that interaction with `SomaticMarker(valence=-0.8, arousal=0.7, label=\"sadness\")` and gives it higher recall priority when the topic comes up again.\n\n### Metrics\n\n| Metric | Definition | Target | Code Path |\n|--------|-----------|--------|-----------|\n| `sentiment_accuracy` | % of labeled test inputs where `detect_sentiment()` returns the expected label | >= 0.75 | `memory/sentiment.py::detect_sentiment()` |\n| `emotional_storage_rate` | % of high-arousal interactions (arousal >= 0.6) that pass the significance gate | >= 0.80 | `memory/attention.py::is_significant()` |\n| `neutral_rejection_rate` | % of low-arousal short filler turns (<12 tokens) that are correctly rejected by gate | >= 0.70 | `memory/attention.py::overall_significance()` |\n| `mood_responsiveness` | After 5 consecutive interactions with same emotion label, does `SoulState.mood` match? | >= 0.80 accuracy | `state/manager.py::on_interaction()` |\n| `emotional_arc_coherence` | In `emotional_rollercoaster` scenario, do stored episodic memories' somatic labels cluster by phase (happy/sad/angry)? | Cluster purity >= 0.60 | `memory/episodic.py` entries |\n| `emotional_recall_bias` | Are high-arousal memories (arousal >= 0.5) retrieved more often than neutral ones for emotionally-charged queries? | Yes — ratio >= 1.5x | `memory/activation.py::compute_activation()` |\n\n### Scenarios\n\n**Scenario EI-1: Sentiment Classification Benchmark**\n\nA labeled corpus of 60 interaction strings, manually categorized into 7 emotion labels\n(excitement, joy, gratitude, curiosity, frustration, sadness, neutral). These are drawn\nfrom the existing `emotional_rollercoaster` turns — use the planted high-emotion turns\nfrom `scenarios.py` as ground truth. Run `detect_sentiment()` on each and check label match.\n\nExample ground truth:\n- \"I GOT PROMOTED!!!\" → `excitement`\n- \"I don't know how to say this... my dog Biscuit passed away yesterday.\" → `sadness`\n- \"I am FURIOUS. Someone hacked my bank account.\" → `frustration`\n- \"Nice weather today, isn't it?\" → `neutral`\n\n**Scenario EI-2: Gate Calibration Test**\n\nFeed 100 interactions through the significance gate: 50 high-emotion (from `emotional_rollercoaster`\nplanted phases), 50 filler (from `_SMALL_TALK` and `_WEATHER_CHAT`). Measure\n`emotional_storage_rate` and `neutral_rejection_rate`.\n\n**Scenario EI-3: Mood State Machine Test**\n\nCreate a soul. Feed 5 consecutive interactions with strong `frustration` label (from\n`angry_turns` in the scenario). Check that `soul.state.mood` becomes `Mood.CONCERNED`.\nThen feed 5 consecutive `excitement` interactions. Check that mood shifts to `Mood.EXCITED`.\nThe EMA smoothing in `state/manager.py` requires ~4 accumulated signals to shift mood,\nso 5 turns is the correct probe depth.\n\n**Scenario EI-4: Emotional Arc Coherence Test**\n\nRun full `emotional_rollercoaster` scenario through `FULL_SOUL` condition. After each\nphase (turns 0-19, 20-39, 40-59), extract the stored episodic memories. For each memory,\nread its `somatic.label`. Check that the dominant label in each phase window matches\nthe expected emotional tone (positive / negative-sad / negative-angry). Cluster purity\nis defined as: majority_label_count / total_memories_in_phase.\n\n### Scoring (0-100)\n\n```\nscore = (sentiment_accuracy * 25)\n      + (emotional_storage_rate * 25)\n      + (neutral_rejection_rate * 20)\n      + (mood_responsiveness * 15)\n      + (emotional_arc_coherence * 15)\n```\n\n### Implementation Notes\n\n- `detect_sentiment()` is a pure function — no soul setup required for EI-1\n- Access episodic memories via `soul._memory._episodic._memories.values()`\n- Access somatic labels via `memory_entry.somatic.label`\n- `soul.state.mood` is accessible via `soul.state.mood`\n- `emotional_recall_bias` requires comparing retrieval frequency of high vs low arousal\n  entries — run recall queries that are emotionally neutral and emotionally charged, compare\n  which memories surface\n\n---\n\n## Dimension 3: Personality Expression\n\n### What It Measures\n\nWhether the OCEAN model actually changes observable system behavior. This is partially\nstructural (does the DNA produce different prompts?) and partially behavioral (does\nsignificance gating reflect the soul's values?). Full LLM output testing is an optional extension.\n\n### Why This Is Tricky\n\nWithout an LLM judge, we cannot verify that a soul with `extraversion=0.9` writes longer,\nwarmer responses than one with `extraversion=0.1`. What we CAN verify is that:\n1. The system prompt accurately encodes the personality and communicates it to an LLM\n2. The significance gate is sensitive to the soul's values (goal_relevance dimension)\n3. The self-model respects personality seeding\n\n### Metrics\n\n| Metric | Definition | Target | Code Path |\n|--------|-----------|--------|-----------|\n| `prompt_fidelity` | Does `to_system_prompt()` include all 5 OCEAN traits at their configured values? | 100% fidelity | `dna/prompt.py::dna_to_system_prompt()` |\n| `communication_style_coverage` | Does system prompt include all 4 CommunicationStyle fields (warmth, verbosity, humor, emoji)? | 100% | `dna/prompt.py` |\n| `value_alignment_gap` | For a soul with core_values=[\"empathy\"], goal_relevance scores for emotional turns vs technical turns | Emotional > Technical by >= 0.1 delta | `memory/attention.py::compute_significance()` |\n| `personality_stability` | After 200 interactions, do OCEAN trait values remain unchanged (no unintended drift)? | 100% stable | `soul.dna.personality` |\n| `prompt_differentiation` | System prompt diff between two souls with identical names but opposite OCEAN profiles (high vs low on all traits) | >= 5 distinct lines different | `to_system_prompt()` |\n\n### Scenarios\n\n**Scenario PE-1: Prompt Fidelity Test**\n\nBirth a soul with specific OCEAN values (openness=0.9, conscientiousness=0.2,\nextraversion=0.8, agreeableness=0.3, neuroticism=0.7). Call `to_system_prompt()`.\nParse the output and verify each trait value appears. This is a regression test — it\ncatches any prompt template changes that silently drop personality data.\n\n**Scenario PE-2: Value-Weighted Significance Test**\n\nCreate two souls: Soul-A has `core_values=[\"empathy\", \"compassion\", \"listening\"]`,\nSoul-B has `core_values=[\"efficiency\", \"speed\", \"precision\"]`. Feed both the same\n20 interactions: 10 emotional (grief, celebration, frustration) and 10 technical\n(code review, performance optimization). For each interaction, measure `goal_relevance`\nfrom `compute_significance()`. Soul-A should give higher goal_relevance to emotional\nturns; Soul-B should give higher goal_relevance to technical turns.\n\n**Scenario PE-3: Personality Contrast Test**\n\nBirth two souls: Soul-HIGH (openness=0.9, extraversion=0.9, agreeableness=0.9)\nand Soul-LOW (openness=0.1, extraversion=0.1, agreeableness=0.1). Compare their\nsystem prompts. Count lines that differ. Count tokens that differ. This produces\nthe `prompt_differentiation` metric.\n\n**Scenario PE-4: OCEAN Stability Under Interaction**\n\nRun a 200-turn conversation through `observe()` for a soul with specific OCEAN values.\nAfter each 50 turns, read `soul.dna.personality` and assert all 5 trait values match\nthe initial configuration. This validates that the heuristic observe pipeline does not\naccidentally mutate DNA (only `approve_evolution()` should change traits).\n\n### Scoring (0-100)\n\n```\nscore = (prompt_fidelity * 30)\n      + (communication_style_coverage * 10)\n      + (value_alignment_gap_passes * 25)  # binary: does soul-A score higher on emotional? 0 or 25\n      + (personality_stability * 25)        # binary: no drift = 25\n      + min(10, prompt_differentiation_lines * 2)\n```\n\n### LLM-Enhanced Extension (optional, not required for core SHS)\n\nWhen an LLM is available, feed the two souls from PE-3 the same 10 prompts and\nmeasure response-level differences: average word count, sentiment of response,\nuse of first-person warmth markers (\"I understand\", \"that sounds hard\"). This produces\n`output_differentiation_score` which is reported separately and not included in\ncore SHS to keep it LLM-free.\n\n---\n\n## Dimension 4: Bond / Relationship\n\n### What It Measures\n\nWhether the bond system creates a real arc — early interactions feel different from\ndeep, long-term ones. Bond strength should grow at the documented logarithmic rate,\nwith positive interactions strengthening faster than neutral ones.\n\n### Bond System Reference\n\nFrom `bond.py`:\n- Initial bond: 50.0\n- Growth formula: `effective = amount * (remaining / 100.0)` — logarithmic decay\n- Positive interactions: `amount = 1.0 + somatic.valence` (up to 2.0 at max positivity)\n- Neutral interactions: `amount = 0.5`\n- Documented milestone: reaching 99 from 50 takes ~460 interactions at amount=1.0\n\n### Metrics\n\n| Metric | Definition | Target |\n|--------|-----------|--------|\n| `bond_growth_curve` | Correlation of observed bond_trajectory with expected logarithmic model | Pearson r >= 0.95 |\n| `positive_acceleration` | Ratio of bond_strength after 100 positive turns vs 100 neutral turns | >= 1.15x |\n| `bond_tier_progression` | Does bond cross defined tier thresholds in expected order at expected rates? | All tiers hit in correct order |\n| `bond_milestone_accuracy` | Starting at 50, after N documented interactions, does bond match formula prediction within ±2 pts? | Within tolerance for N=50, 100, 200 |\n| `bond_asymmetry` | Does weaken(amount=0.5) reduce bond more than strengthen(amount=0.5) raises it at high bond levels? | Yes (documented behavior) |\n\n### Bond Tier Definitions\n\nDefine 5 tiers based on bond_strength for reporting:\n\n| Tier | Range | Label |\n|------|-------|-------|\n| 0 | 0-20 | Stranger |\n| 1 | 21-40 | Acquaintance |\n| 2 | 41-60 | Familiar |\n| 3 | 61-80 | Friend |\n| 4 | 81-100 | Bonded |\n\n### Scenarios\n\n**Scenario BD-1: Growth Curve Validation**\n\nCreate a soul. Run 200 interactions with `somatic.valence = 0` (neutral). Record bond\ntrajectory after each interaction. Fit the trajectory to the expected model:\n`bond(n) = 100 * (1 - e^(-k*n))` for some k. Pearson r between observed and\npredicted is `bond_growth_curve`.\n\n**Scenario BD-2: Valence Acceleration**\n\nCreate two souls (same config). Feed Soul-A 100 interactions with `somatic.valence = 0.8`\n(strongly positive, like promotion announcement). Feed Soul-B 100 neutral interactions\n(`somatic.valence = 0`). Measure final bond strength for both. `positive_acceleration`\nis `soul_A.bond_strength / soul_B.bond_strength`.\n\n**Scenario BD-3: Milestone Accuracy**\n\nStarting at bond=50, compute expected values at N=50, 100, 200, 300 using the formula\n`strengthen(amount=1.0)` repeated N times. Compare to actual. Check each is within ±2 pts.\n\n**Scenario BD-4: Long-Horizon Bond in Scenario Context**\n\nExtract bond trajectory from `emotional_rollercoaster` run in full_soul condition\n(already collected by runner.py `bond_trajectory` field). Check that bond strength\nafter the happy phase (turns 0-19) is higher than after the angry phase (turns 40-59),\ndemonstrating that somatic valence influences bond growth rate in realistic scenarios.\n\n### Scoring (0-100)\n\n```\nscore = (bond_growth_curve_r * 40)        # correlation fidelity\n      + (positive_acceleration_passes * 20)  # binary: >= 1.15x\n      + (milestone_accuracy_all_pass * 25)   # binary: all 3 milestones within tolerance\n      + (long_horizon_bond_phase_ordering * 15)  # binary: correct ordering\n```\n\n---\n\n## Dimension 5: Self-Model\n\n### What It Measures\n\nWhether the soul builds an accurate self-concept from experience — the Klein self-model\n(`memory/self_model.py`). After 100 coding conversations, does the soul know it's a\ntechnical helper? Can it correctly describe itself? Does it NOT identify as a creative\nwriter?\n\n### Metrics\n\n| Metric | Definition | Target |\n|--------|-----------|--------|\n| `domain_accuracy` | After N on-topic interactions, is the top self_image domain the expected one? | >= 0.85 accuracy |\n| `domain_emergence_speed` | How many on-topic interactions before top domain confidence crosses 0.4? | <= 20 interactions |\n| `domain_specificity` | Cross-contamination: after 50 coding interactions, is `emotional_companion` NOT in top-3 domains? | Yes (no contamination) |\n| `confidence_growth_curve` | Does domain confidence follow the documented diminishing returns formula? | Pearson r >= 0.90 |\n| `relationship_note_extraction` | After stating \"My boss is Sarah Chen\", does `relationship_notes['user']` contain the fact? | Binary: yes/no |\n| `domain_keyword_expansion` | After 10 technical interactions, has the `technical_helper` domain's keyword count grown? | Yes |\n\n### Formula Reference\n\nFrom `self_model.py`:\n```\nconfidence = min(0.95, 0.1 + 0.85 * (1 - 1 / (1 + evidence_count * 0.1)))\n```\n\nAt evidence_count=10: confidence ≈ 0.48\nAt evidence_count=20: confidence ≈ 0.63\nAt evidence_count=50: confidence ≈ 0.81\nAt evidence_count=100: confidence ≈ 0.89\n\n### Scenarios\n\n**Scenario SM-1: Domain Classification Accuracy**\n\nCreate 4 souls, each fed 50 single-topic interactions:\n- Soul-TECH: 50 interactions from a coding scenario (Python, debugging, APIs)\n- Soul-WRITE: 50 interactions about creative writing (stories, essays, plots)\n- Soul-EMO: 50 interactions about emotional support (feelings, stress, relationship)\n- Soul-MIXED: 50 interactions alternating between all three domains\n\nFor Soul-TECH: top domain should be `technical_helper`.\nFor Soul-WRITE: top domain should be `creative_writer`.\nFor Soul-EMO: top domain should be `emotional_companion`.\nFor Soul-MIXED: no single domain should dominate (no domain > 0.7 confidence).\n\n`domain_accuracy` = fraction of single-topic souls that hit expected top domain.\n\n**Scenario SM-2: Emergence Speed Test**\n\nFeed Soul-TECH interactions one at a time and check after each interaction whether\nthe `technical_helper` domain has confidence >= 0.4. The turn where it crosses 0.4\nis the `emergence_turn`. `domain_emergence_speed = emergence_turn` (lower is better).\n\n**Scenario SM-3: Cross-Domain Isolation Test**\n\nFeed Soul-TECH 50 coding interactions. Read `soul.self_model.self_images`. Verify that\n`emotional_companion` either does not appear, or appears with confidence < 0.3.\nThis tests that the stop-word filtering and domain scoring correctly prevent contamination.\n\n**Scenario SM-4: Relationship Note Test**\n\nCreate a soul. Feed it the interaction: user=\"My boss is named Sarah Chen.\",\nagent=\"Good to know! It's helpful to understand your work relationships.\"\nCall `soul.self_model.relationship_notes` and check for \"Sarah Chen\" or \"Works at\"\ntype content. This maps directly to the `update_from_interaction()` pattern-matching\ncode in `self_model.py`.\n\n### Scoring (0-100)\n\n```\nscore = (domain_accuracy * 35)\n      + max(0, 25 - domain_emergence_speed)   # 25 pts at emergence <= 0, 0 pts at >= 25\n      + (domain_specificity_passes * 20)       # binary\n      + (confidence_curve_r * 10)\n      + (relationship_note_extraction * 10)    # binary\n```\n\n---\n\n## Dimension 6: Identity Continuity\n\n### What It Measures\n\nExport a `.soul` file, reimport it on a \"new platform\" (new Soul instance). Is the\nidentity preserved exactly? This is the core portability claim of the protocol.\n\n### What \"Same Soul\" Means\n\nAfter export/import, the following must be bit-for-bit or semantically identical:\n\n- `soul.did` (cryptographic identity — must match exactly)\n- `soul.name`\n- `soul.dna.personality` (all 5 OCEAN traits within floating-point epsilon)\n- `soul.dna.communication` (all 4 fields)\n- `soul.bond.bond_strength` (within ±0.01)\n- `soul.bond.interaction_count`\n- `soul.memory_count` (total memories across all tiers)\n- Top-5 semantic facts (by importance) are all present after import\n- Top-3 recall results for 5 standard queries are the same before and after export\n- `soul.self_model.get_active_self_images()` — same top domain, within ±0.05 confidence\n\n### Metrics\n\n| Metric | Definition | Target |\n|--------|-----------|--------|\n| `identity_hash_match` | DID, name, born timestamp all match after export/import | 100% |\n| `dna_fidelity` | All 5 OCEAN traits within epsilon=0.001 after round-trip | 100% |\n| `bond_fidelity` | bond_strength and interaction_count match | 100% |\n| `memory_count_fidelity` | Total memory count matches after round-trip | 100% |\n| `recall_consistency` | For 5 standard queries, top result is same before and after export | >= 0.80 |\n| `self_model_fidelity` | Top self_image domain and confidence within ±0.05 | >= 0.90 |\n| `incarnation_chain_integrity` | `reincarnate()` increments `incarnation` and appends old DID to `previous_lives` | 100% |\n\n### Scenarios\n\n**Scenario IC-1: Basic Export/Import Round-Trip**\n\nBirth a soul. Run 50 `observe()` calls with the `life_updates` turns. Export to `.soul`\nbytes. Birth a new Soul from those bytes using `Soul.awaken()`. Compare all fields above.\n\n**Scenario IC-2: Multi-Format Round-Trip**\n\nRun the same soul through 4 export formats:\n1. `.soul` file (zip archive via `pack_soul()`)\n2. `soul.json` (SoulConfig JSON)\n3. `soul.yaml` (SoulConfig YAML)\n4. `.soul/` directory (via `save_local()`)\n\nFor each format, check identity hash match and recall consistency with the same 5 queries.\n\n**Scenario IC-3: Incarnation Lineage Test**\n\nBirth Soul-1. Run 20 interactions. Reincarnate to Soul-2. Run 20 more interactions.\nReincarnate to Soul-3. Verify:\n- `soul_3.identity.incarnation == 3`\n- `soul_3.identity.previous_lives == [soul_1.did, soul_2.did]`\n- Soul-3 has access to Soul-1's memories (carried through reincarnation)\n- Soul-3's DID is different from Soul-1's and Soul-2's\n\n**Scenario IC-4: Recall Consistency Across Platforms**\n\nAfter export/import, run the same 10 recall queries that were run before export.\nFor each query, verify the top-1 result content is identical. This proves that\nmemory search is deterministic and export-lossless.\n\n### Scoring (0-100)\n\n```\nscore = (identity_hash_match * 25)         # binary\n      + (dna_fidelity * 20)                 # binary\n      + (bond_fidelity * 15)                # binary\n      + (memory_count_fidelity * 10)        # binary\n      + (recall_consistency * 20)           # fraction\n      + (incarnation_chain_integrity * 10)  # binary\n```\n\nIf all binary checks pass, minimum score is 80 before recall_consistency.\n\n---\n\n## Dimension 7: Portability\n\n### What It Measures\n\nThe same `.soul` file works with different cognitive engines. Soul identity (personality,\nmemories, bond) is engine-agnostic — it lives in the soul, not in the LLM.\n\n### Testable Without Multiple Real LLMs\n\n`CognitiveEngine` is a Protocol in `runtime/cognitive/engine.py`. We have:\n- `HeuristicEngine` — available without API keys\n- Mock engines that can be constructed in tests\n\nThe key claim: swapping engines does not change the soul's stored data, system prompt,\nor recall behavior.\n\n### Metrics\n\n| Metric | Definition | Target |\n|--------|-----------|--------|\n| `system_prompt_engine_independence` | `to_system_prompt()` produces identical output regardless of which engine is attached | 100% identical |\n| `recall_engine_independence` | Recall results are identical with HeuristicEngine vs no engine | 100% identical |\n| `observe_pipeline_stability` | After `observe()` with HeuristicEngine, soul state matches expected values | All assertions pass |\n| `engine_swap_continuity` | Birth with HeuristicEngine, save, awaken with no engine, save, awaken with HeuristicEngine — soul state unchanged | Memory count identical |\n\n### Scenarios\n\n**Scenario PT-1: System Prompt Independence**\n\nBirth Soul-A with `engine=HeuristicEngine()`. Birth Soul-B with `engine=None`. Same name,\nsame OCEAN values, same persona. Call `to_system_prompt()` on both. They must be identical.\n(The system prompt is generated from DNA and core memory, neither of which touches the engine.)\n\n**Scenario PT-2: Recall Independence**\n\nCreate a soul. Store 20 memories manually via `soul.remember()`. Run 5 recall queries with\n`engine=HeuristicEngine()` attached. Detach engine (create new soul with same saved data\nbut `engine=None`). Run same 5 queries. Results must be identical. (Recall uses BM25 +\nACT-R activation — no LLM involved.)\n\n**Scenario PT-3: Engine Swap Continuity**\n\nBirth soul with HeuristicEngine. Observe 30 interactions. Save. Awaken with no engine.\nObserve 10 more interactions. Save. Awaken with HeuristicEngine. Verify:\n- memory_count equals what it was before the last save (no data loss from swap)\n- bond_strength matches last recorded value\n- top self_image domain unchanged\n\n### Scoring (0-100)\n\n```\nscore = (system_prompt_engine_independence * 35)   # binary\n      + (recall_engine_independence * 35)           # binary\n      + (engine_swap_continuity * 30)               # binary\n```\n\nThis dimension is pass/fail in nature — portability either works or it doesn't.\nPartial credit only if one specific engine combination fails.\n\n---\n\n## Soul Health Score (SHS)\n\n### Composite Formula\n\n```\nSHS = (D1_memory   * 0.20)\n    + (D2_emotion  * 0.20)\n    + (D3_personality * 0.15)\n    + (D4_bond     * 0.15)\n    + (D5_self_model * 0.15)\n    + (D6_continuity * 0.10)\n    + (D7_portability * 0.05)\n```\n\nEach D-score is 0-100.
