## Purpose

`parse_soul_yaml()` handles the YAML soul configuration format — typically used when a soul is defined in a machine-readable but human-editable file, or when the soul config is generated by tooling that emits YAML rather than JSON.

## Implementation

```python
def parse_soul_yaml(content: str) -> SoulConfig:
    data = yaml.safe_load(content)
    return SoulConfig.model_validate(data)
```

Unlike `parse_soul_json()`, this is a two-step process: YAML deserialization first, then Pydantic validation. This is necessary because Pydantic v2's `model_validate_json()` only accepts JSON syntax — there is no `model_validate_yaml()`. The intermediate dict from `yaml.safe_load()` is passed to `model_validate()` which handles type coercion and validation.

## Why `safe_load` and Not `full_load`?

`yaml.safe_load()` is mandatory for untrusted input. YAML's `full_load` (or the older `yaml.load()` without a Loader) can deserialize arbitrary Python objects via YAML tags like `!!python/object`. A soul config file from an untrusted source (e.g., downloaded from a repository) could exploit this to execute arbitrary code. `safe_load` rejects all tags beyond basic YAML types.

This is especially important in Soul Protocol because users can share `.soul` files — which embed YAML-derived config — across platforms.

## Error Contract

Two distinct exception types flow from this function:

- `yaml.YAMLError`: The string is not valid YAML (syntax error).
- `pydantic.ValidationError`: The YAML was valid but the resulting data doesn't conform to `SoulConfig` (missing required fields, wrong types, etc.).

Separating these allows callers to produce targeted error messages: "This file has a YAML syntax error on line 12" versus "This soul config is missing the required `name` field".

## Comparison to JSON Parser

The JSON parser uses `model_validate_json()` which is faster because it parses JSON directly into Pydantic's internal representation. The YAML parser is two steps because of the missing Pydantic YAML pathway — it produces the same validated `SoulConfig` output but with more intermediate allocations.

## Known Gaps

- YAML anchors and aliases (`&anchor`, `*alias`) work through `safe_load` and can be used to define reusable personality or DNA blocks in a soul config file. However, Soul Protocol has no documentation or validation for this pattern, and circular aliases would cause `safe_load` to raise an error rather than detect the cycle gracefully.
- No schema version field in the YAML format, same as the JSON parser.

## YAML vs. JSON for Soul Configs

YAML offers advantages for hand-authored soul files: multiline strings without escape sequences, comments for documenting design intent, and a more readable syntax for nested structures like personality traits:

```yaml
# Soul configuration for Luna
name: Luna
personality:
  openness: 0.8      # Curious and creative
  conscientiousness: 0.6  # Organized but flexible

core_memory:
  persona: |
    Luna is a creative writing partner who thrives on
    imaginative collaboration and narrative exploration.
```

This readability is why `soul.yaml` is often preferred over `soul.json` for souls authored by human developers rather than generated by code.

## Testing Pattern

Like the JSON parser, `parse_soul_yaml()` is a pure function that is easy to test exhaustively:

```python
def test_invalid_yaml_raises():
    with pytest.raises(yaml.YAMLError):
        parse_soul_yaml("name: [unclosed")

def test_invalid_schema_raises():
    with pytest.raises(pydantic.ValidationError):
        parse_soul_yaml("name: 123")  # name must be str
```

The two-exception contract makes it straightforward to write tests that distinguish parse failures from schema failures.
