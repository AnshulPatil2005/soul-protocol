## Overview

`FileStorage` is the primary persistence backend for production soul use. It maps each soul to a directory keyed by its sanitized soul ID and writes three core files: `soul.json` (full `SoulConfig`), `dna.md` (human-readable personality), and `state.json` (current state snapshot). The full-persistence functions extend this with a `memory/` subdirectory holding seven JSON files for all memory tiers.

## Directory Layout

```
~/.soul/
  did_soul_name-hash/
    soul.json         # full SoulConfig
    dna.md            # human-readable DNA markdown
    state.json        # current SoulState snapshot
    memory/
      core.json
      episodic.json
      semantic.json
      procedural.json
      graph.json
      self_model.json
      general_events.json
```

The three-file root layout is intentional: `soul.json` is the authoritative config, `dna.md` is for humans to read and debug, and `state.json` is a quick snapshot of current mood and energy that tools can read without parsing the full config.

## Atomic Write Pattern

`save_soul_full` never writes directly to the soul directory. It builds all output in a temp directory first, then moves it into place:

```python
with tempfile.TemporaryDirectory(dir=base) as tmp:
    tmp_dir = Path(tmp) / soul_id
    tmp_dir.mkdir()
    _write_soul_files(tmp_dir, config, memory_data)
    if soul_dir.exists():
        shutil.rmtree(soul_dir)
    shutil.move(str(tmp_dir), str(soul_dir))
```

This prevents a partially-written soul directory if the process is killed mid-write. Without this guard, a crash after writing `soul.json` but before writing `memory/episodic.json` would leave the soul in an inconsistent state — the config would load but memories would silently be empty. The temp directory is placed in the same `base` parent so that `shutil.move` can use an atomic rename on most filesystems rather than a cross-device copy.

## Cross-Platform Soul ID Safety

DID strings contain colons (`did:soul:name-hash`), which are illegal in Windows paths. `_safe_soul_id()` replaces colons with underscores and rejects IDs containing `..`, `/`, or `\`:

```python
def _safe_soul_id(config: SoulConfig) -> str:
    soul_id = config.identity.did or config.identity.name
    if ".." in soul_id or "/" in soul_id or "\\" in soul_id:
        raise ValueError(f"Soul ID contains unsafe path characters: {soul_id}")
    return soul_id.replace(":", "_")
```

Path traversal characters are caught before any filesystem operation, preventing a crafted soul ID from escaping the base directory. This is a defense-in-depth measure — in practice, soul IDs are generated by the runtime, not user-supplied — but the check prevents a compromised template or malicious import from writing outside `~/.soul/`.

## Deprecation: save_soul

The original `save_soul()` convenience function is preserved but deprecated. It silently dropped all memory tier data — only `soul.json` and `dna.md` were written. A soul that appeared to save successfully would lose all memories on the next load. `save_soul_full()` corrects this and is the only recommended path for new code.

## save_soul_flat: Project-Folder Variant

`save_soul_flat` writes without soul-ID nesting, used for `.soul/` project folders where the directory itself is the soul. It uses the same atomic temp-dir approach but moves individual files rather than the whole directory, to preserve existing folder structure.

## Known Gaps

`save_soul_flat` moves files individually — a crash between file moves could leave mixed old/new data in the target directory. This is lower risk for project-folder souls (frequently overwritten) but is not equivalent to the full-directory-move guarantee of `save_soul_full`.
